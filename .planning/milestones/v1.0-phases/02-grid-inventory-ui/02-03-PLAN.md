    Rewrite scripts/world.gd:
    ```gdscript
    extends Node2D

    @onready var player: Player = $Player
    @onready var inventory_ui: Control = $CanvasLayer/InventoryUI
    @onready var rejection_label: Label = $CanvasLayer/RejectionLabel
    @onready var fade_timer: Timer = $CanvasLayer/RejectionLabel/FadeTimer


    func _ready() -> void:
        inventory_ui.set_inventory(player.inventory)
        player.inventory.insert_rejected.connect(_on_insert_rejected)
        fade_timer.timeout.connect(_on_fade_timer_timeout)


    func _on_insert_rejected() -> void:
        rejection_label.modulate.a = 1.0
        fade_timer.start()


    func _on_fade_timer_timeout() -> void:
        var tw := create_tween()
        tw.tween_property(rejection_label, "modulate:a", 0.0, 0.4)


    func _on_detection_area_home_body_entered(_body):
        BackgroundMusic.play_audio("home")
    ```

    Key constraints:
    - rejection_label starts with modulate.a = 0.0 (set in .tscn) and visible = true. Show by setting modulate.a = 1.0.
    - Do NOT use rejection_label.visible = false/true — fade via modulate.a only.
    - The existing _on_detection_area_home_body_entered must be preserved.
    - set_inventory() must be called AFTER player is ready. Since world._ready() runs after children's _ready(), player.inventory will be the cloned runtime copy at this point — correct per Pattern 5 in RESEARCH.md.

  <files>scripts/collectable.gd</files>
  <action>
    The current collectable.gd collect() silently discards the rejection case (only calls queue_free on success).
    The insert_rejected signal on Inventory (emitted inside insert()) is already connected to _on_insert_rejected() in world.gd.
    Therefore collectable.gd does NOT need to call any HUD method directly — the signal from Inventory handles it.

    However, the current collect() method passes `item` (the Item resource), not `amount`:
    ```gdscript
    var success = collector.collect(item)
    ```
    And player.collect() calls `inventory.insert(item)` which returns 0 on full success.
    The insert_rejected signal will fire automatically from within insert() if weight blocks insertion.
    No change to collectable.gd is strictly required for the signal path.

    BUT: there is a UX gap — when collect() returns false (insertion partially failed or fully failed), the collectable currently stays in the world (queue_free is not called). This is correct behavior. Verify this is still the case and no change is needed.

    Also verify: player.collect() currently returns `inventory.insert(item) == 0`. This means it returns true only when ALL items inserted. Since collectables pass 1 item at a time, this is: true if the 1 item was inserted, false if rejected. The insert_rejected signal only fires when weight blocks ALL insertion (remaining == amount). For a single item, this means: if the item is too heavy, insert_rejected fires AND collect() returns false. Correct.

    Make one defensive addition to collectable.gd to avoid silent failure: add a comment clarifying the rejection path, and ensure the collect() function body is clean:

    ```gdscript
    func collect():
        assert(collector.has_method("collect"), "Player script should have a collect method.")
        var success = collector.collect(item)
        if success:
            queue_free()
        # If not success: inventory.insert() already emitted insert_rejected signal.
        # The HUD label is handled by world.gd listening to that signal.
    ```

    This is a documentation-only change — the logic does not change. If the file already matches this pattern exactly (minus the comment), the comment addition is still valuable for clarity. Apply the change.
